## Goal
Build `docpilot-mcp`, a TypeScript MCP server that exposes 4 tools (`query_docs`, `get_changelog`, `search_examples`, `resolve_method`) so any AI coding agent can fetch live, version-accurate documentation and code examples for any npm or PyPI package (plus major frameworks with dedicated doc sites) without hallucinating stale method signatures.

- Node v24.15.0 and npm 11.12.1 are preinstalled; no package manager is pinned yet.

- The user-facing "agent" is an AI coding assistant (Claude Code, Cursor, Codex, etc.) that talks to the MCP server over stdio. The MCP server itself does the fetching, parsing, and indexing.
- Source of truth is the package's official docs site or GitHub repo. Third-party aggregators (npmjs.com READMEs, random blogs) are explicitly out of scope as a source of truth.
- The server runs locally on the developer's machine. Caching must persist across restarts to keep `query_docs` and `resolve_method` sub-second on the second call.
- For browser-rendered doc sites (Stripe, Vercel, Cloudflare, etc.), Playwright is required. For static doc sites, cheerio is sufficient.

## Acceptance Criteria

- AC-1: Project bootstrap is reproducible and the server is runnable.
  - `package.json` declares `"type": "module"`, a `build` script that produces `dist/`, a `start` script that runs `node dist/index.js`, and a `dev` script that runs `tsx src/index.ts`.
  - `tsconfig.json` uses `"strict": true`, `"target": "ES2022"`, `"module": "NodeNext"`, `"moduleResolution": "NodeNext"`, and `"outDir": "dist"`.
  - `npm install` then `npm run build` then `npm start` exits with code 0 when stdin is closed and prints a single line to stderr: `docpilot-mcp ready (tools=4)`.
  - `node -e "import('./dist/index.js')"` resolves without throwing under ESM.

- AC-2: MCP server registers exactly 4 tools with the right names, descriptions, and input schemas.
  - Tools registered: `query_docs`, `get_changelog`, `search_examples`, `resolve_method`.
  - Each tool has a JSON Schema input that requires the documented fields, has no additional `any`-typed properties, and rejects missing required fields with a clear MCP `InvalidRequest` error.
  - The tool list can be enumerated by running `npm run inspect` (a dev script that calls `server.listTools()`) and prints the 4 names plus a one-line description for each.

- AC-3: `query_docs` returns a version-accurate, cited, ranked context block.
  - Required input: `{ "package": "stripe", "version": "5.0.0", "question": "how do I paginate cursor results" }`.
  - Output is a single object: `{ "package", "version", "answer_markdown", "sources": [{ "url", "section", "snippet", "score" }] }`.
  - `answer_markdown` is at most 2000 tokens and always ends with a `Sources:` list linking to each `sources[].url`.
  - The top result's `snippet` contains a literal phrase from the package's own docs (e.g. `auto_pagination_iter`), verified by a regex assertion in `tests/`.
  - If a `version` not yet cached is requested, the server fetches and caches it; subsequent calls for the same `version` must be served from cache (verified by a `Cache-Control`-style header-equivalent log line `cache=hit`).
  - Works for at least these packages end-to-end against live docs: `stripe` (JS SDK v5), `requests` (PyPI), `next` (Next.js 15). Recorded in `docs/references/verified-packages.md`.

- AC-4: `get_changelog` returns the latest changelog entries for a package.
  - Required input: `{ "package": "stripe", "version": "latest" }` (version optional, defaults to `latest`).
  - Supports npm via `https://registry.npmjs.org/<pkg>` and PyPI via `https://pypi.org/pypi/<pkg>/json`.
  - Returns `{ "package", "ecosystem", "version", "entries": [{ "version", "date", "summary" }] }` with the 10 most recent entries.
  - When the registry does not include a changelog field, falls back to fetching `CHANGELOG.md` from the package's default branch on GitHub (inferred from the registry `repository` field) and parses top-level headings.
  - Verified for both ecosystems by `tests/changelog.test.ts` using recorded fixture responses under `test/fixtures/registry/`.

- AC-5: `search_examples` returns real code examples from official sources only.
  - Required input: `{ "package": "stripe", "version": "5.0.0", "query": "create a customer" }`.
  - Searches the official `examples/` directory and the `README.md` of the GitHub repo resolved from the registry. Does **not** call Stack Overflow, blogs, or third-party aggregators.
  - Returns up to 10 `{ "code", "path", "url", "language" }` blocks; `language` is inferred from file extension or fenced-code info string.
  - Each returned `code` block must compile or parse as valid syntax for the declared `language` (verified by running `node --check` for `.js`/`.ts` snippets and a Python AST parse for `.py` snippets in `tests/examples.test.ts`).

- AC-6: `resolve_method` returns the current signature for a method.
  - Required input: `{ "package": "stripe", "version": "5.0.0", "method": "customers.create" }`.
  - Output: `{ "package", "version", "method", "signature", "params": [{ "name", "type", "required", "description" }], "returns", "source": { "url", "path", "line" } }`.
  - For TypeScript/JS packages: signatures are extracted from the `.d.ts` file in the tarball; the `source.path` is the relative path inside the tarball, and `source.line` is the 1-indexed line of the declaration.
  - For Python packages: signatures are extracted from the wheel's `.pyi` stubs when present, otherwise from the wheel's `.py` source using `ast` to find `def` nodes; returns `source.line`.
  - Returns a `MethodNotFound` MCP error if the method does not exist in the requested version (does not silently fall back to a different version).

- AC-7: A local vector index is built once per `(package, version)` and reused.
  - On first call to `query_docs` for a new `(package, version)`, the server downloads the docs, extracts text chunks (~500 tokens, 50-token overlap), embeds them with a local model, and stores them in `~/.cache/docpilot-mcp/index/<ecosystem>/<package>/<version>/`.
  - Re-issuing the same call returns within 500 ms locally for a 500-chunk index.
  - The cache directory layout, content-addressable hash, and schema for `chunks.jsonl` are documented in `docs/cache-format.md` and validated by `tests/cache.test.ts`.

- AC-8: Static and JS-rendered doc sites are both supported.
  - Default fetch path uses `cheerio` for static HTML.
  - If the rendered HTML contains `<noscript>` placeholders, a `data-reactroot` mount point, or fewer than 50 characters of body text after cheerio parsing, the server falls back to Playwright with a 10-second timeout, waits for `networkidle`, and re-parses the page.
  - This logic lives in `src/sources/fetchPage.ts` and is unit-tested with two recorded fixtures: `test/fixtures/static.html` and `test/fixtures/spa.html`.
  - Playwright launches with a single `chromium` browser instance reused across calls (singleton in `src/browser/playwrightPool.ts`).

- AC-9: Caching, rate-limiting, and error handling are operator-friendly.
  - All outbound HTTP requests go through `src/net/httpClient.ts`, which enforces a 10 req/s per-host rate limit, a 3-retry exponential backoff (250ms, 1s, 4s) on 5xx and network errors, and a hard 15-second timeout per request.
  - Disk cache writes are atomic (write to `*.tmp` then rename).
  - All tool errors are returned as MCP `isError: true` results with a stable machine-readable `code` field (`E_NOT_FOUND`, `E_RATE_LIMIT`, `E_UPSTREAM`, `E_INVALID_INPUT`) and a human-readable `message`.

- AC-10: Test suite and developer ergonomics are complete.
  - `npm test` runs `vitest run` and exits 0.
  - Test coverage: at least one unit test per tool handler, one integration test for each of AC-3, AC-4, AC-5, AC-6 using a recorded fixture, and one end-to-end test that boots the MCP server and calls all 4 tools via `Client` against the in-process `StdioServerTransport`.
  - `README.md` documents install, run, the 4 tools with one example per tool, and how to point a Claude/Cursor/Codex client at the server (`claude mcp add docpilot -- npx -y docpilot-mcp` once published, or `node /abs/path/to/dist/index.js`).
  - `docs/references/verified-packages.md` lists the packages verified end-to-end, with example questions and the snippet of the answer that proves the version was honored.

**Risk controls:**
- **Stale model cache.** First model load downloads ~25 MB. `src/index/embed.ts` logs progress to stderr and falls back to a hash-only lexical index (`src/index/lexical.ts`) if the model fails to load, so the server still works offline after the first failure.
- **HuggingFace unreachable.** On `fetch` failure for the model, log a `WARN` and switch to lexical search for that session; never crash the server.
- **Playwright not installed.** `playwrightPool.ts` detects missing browser binary and disables the fallback, returning a clear `E_UPSTREAM` with the install command (`npx playwright install chromium`).
- **MCP protocol drift.** Pin `@modelcontextprotocol/sdk` to a minor version and add a `tests/protocol-smoke.test.ts` that calls `initialize` and `tools/list` against a live `Server`.

- `agent-browser open <url> && agent-browser snapshot > docs/references/<name>.html` — evidence capture for AC-3/AC-5.

## Out of Scope
- A hosted/SaaS version of the server. Only the local stdio MCP server is in scope.
- Authentication for private/paid doc sites. The server only fetches public, unauthenticated pages.
- Crawling third-party aggregators (npmjs.com READMEs, MDN mirrors, Stack Overflow, dev.to, etc.). Only official package registries, official GitHub repos, and the package's own docs site are used.
- Generating documentation that does not exist in the source. The server is a fetcher + ranker, not a writer.
- Real-time index synchronization (webhooks, polling). The cache is refreshed on first access after the index reports `stale=true`; explicit background refreshers are deferred.
- Non-npm/PyPI ecosystems (Cargo, Go modules, RubyGems, Maven). The architecture leaves room for them, but only npm and PyPI ship in v0.1.0.
- A web UI. The server is consumed by an MCP client (Claude Code, Cursor, etc.) only.
- Telemetry, analytics, or any outbound call to a docpilot-controlled endpoint. The server makes no calls to anything other than registries, GitHub, and the package's own docs domain.
